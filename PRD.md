# 個人履歷網站 — 最小需求文件(PRD)

> 定案日期:2026-07-02(經 12 題拷問式訪談收斂)

# 要解決的問題

求職(軟體/AI 相關職缺)需要一份隨時可分享、可投遞的履歷:網路連結給 LinkedIn 與求職信用,PDF 應付傳統投遞——但兩者內容必須永遠一致,不能改一處漏一處。

# 解法(高層次)

以一份 `resume.yaml` 作為**唯一資料源**(欄位對齊 JSON Resume 標準、預留 `{zh, en}` 雙語結構、標記 public/private 個資分級),用 Next.js 靜態輸出成:

1. **一頁式中文履歷網站**(設計感/動效風、RWD、深淺色),部署到 GitHub Pages(一般 repo + 子路徑,需設 basePath)
2. **A4 / ATS 友善的 PDF**:由專用 `/print` 頁面經 Playwright 在 GitHub Actions 自動輸出,放上網站供下載

改 `resume.yaml` → push → 網站與 PDF 同步更新。個資過濾由生成程式強制執行:電話、地址、生日只進 PDF,公開網站僅露出姓名、Email、城市、社群連結。

第一步工作:解析既有舊履歷檔(PDF/Word)轉成 `resume.yaml`,由本人校對後才開發頁面。

**v1 驗收標準——「可投遞」**:

- [ ] 舊履歷轉入 `resume.yaml` 並校對完成
- [ ] 網站含五區塊:簡介/技能/經歷/專案作品集/學歷(作品集為主角)
- [ ] 動效到位且不犧牲載入速度與可讀性;手機、電腦皆正常
- [ ] PDF 可從網站下載,個資分級生效
- [ ] 已上線於 GitHub Pages,網址可分享

# 使用者故事

- 身為**求職者(網站擁有者)**,我想把一個乾淨的網址放進 LinkedIn 和求職信,以便讓對方一鍵看到我的完整作品集與經歷。
- 身為**求職者**,我想從網站直接下載排版好的 PDF 履歷,以便應付要求附檔的傳統投遞流程,且內容保證與網站一致。
- 身為**求職者**,我只想改一份 `resume.yaml` 就讓網站和 PDF 同步更新,以便日後補經歷時不會改一處漏一處。
- 身為**求職者**,我想讓電話、地址只出現在點對點投遞的 PDF 而不在公開網站上,以便避免個資被搜尋引擎索引後收不回來。
- 身為**看履歷的工程主管/人資**,我想在手機或電腦上快速瀏覽作品集與量化成果,以便在幾分鐘內判斷是否邀約面試。

# 不做什麼(Out of Scope)

- **英文版內容**(理由:目前投遞目標以中文市場為主;schema 已預留 `{zh, en}` 欄位,日後補英文內容即可上線,不需重構——現在寫英文是把力氣花在還沒有的需求上)
- **履歷產生器/多版本客製工具**(理由:這版只服務「我的一份履歷」;做成通用工具會讓範圍暴增數倍,且投遞時的微調用改 YAML 就能應付)
- **自訂網域**(理由:GitHub Pages 子路徑網址已可分享;網域是錢和 DNS 設定的獨立決策,隨時可加,不阻塞上線)
- **後台/CMS 編輯介面**(理由:唯一使用者是我自己,直接編輯 `resume.yaml` + git push 就是最簡工作流;做 UI 是純額外成本)
- **部落格、文章區塊**(理由:履歷網站的驗收標準是「可投遞」;部落格是另一個產品,混進來會無限延後上線)
- **瀏覽數據分析、聯絡表單、留言等動態功能**(理由:純靜態輸出是部署與維護成本最低的架構,也是搬家零成本的前提;聯絡方式用 Email 連結即可)
- **多套佈景主題/版型切換**(理由:一套動效風做到精緻,勝過三套做一半;風格調整屬於迭代,不屬於功能)
